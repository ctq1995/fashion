# 迷你播放器贴边自动收起设计

## 背景
当前迷你播放器已经是独立的 Tauri 窗口，具备基础播放控制、置顶、隐藏与回到主窗口能力，界面位于 [MiniPlayerApp.vue](src/MiniPlayerApp.vue)，窗口创建与生命周期管理位于 [lib.rs](src-tauri/src/lib.rs)。

现在需要补充一套更符合桌面悬浮窗习惯的行为：当迷你播放器被拖到屏幕边缘附近时，可以吸附到边缘；在用户松手后，如果窗口仍停留在边缘附近，则经过一段停留时间自动收起到屏幕边缘，只保留一小条可见区域；鼠标再次靠近时，窗口重新展开。

这个能力本质上是原生窗口几何与鼠标命中的问题，不适合主要放在 WebView 前端层完成。前端只能方便地表达 UI 状态，但很难稳定感知原生窗口拖拽结束、监视器工作区边界、离屏偏移量和屏幕级悬停判定。因此本次采用 Rust 原生窗口管理为主、前端最小配合的方案。

## 目标

### 主要目标
- 支持迷你播放器向四个边缘吸附：left、right、top、bottom
- 当用户拖动结束并且窗口足够靠近边缘时，窗口吸附到最近边缘
- 吸附后等待约 1 秒，再自动收起到边缘外
- 收起后只露出一小条边缘，作为可再次唤出的热区
- 鼠标靠近该热区时，窗口完整展开回吸附前的位置
- 鼠标离开已展开窗口后，如果窗口仍保持贴边状态，则再次进入延时收起流程
- 保持迷你播放器现有播放控制、始终置顶、隐藏等能力不变

### 非目标
- 不为主窗口添加同样的贴边自动收起行为
- 不在第一版增加设置面板开关、延时配置或露出厚度配置
- 不实现角落双边同时吸附，只选择最近的一条边
- 不处理复杂的多窗口联动动画
- 不引入与播放状态无关的新全局前端 store

## 设计原则
- 窗口几何、吸附、收起、展开、悬停检测放在 Rust 层统一管理
- 前端只处理展示与必要的状态同步，不承担核心窗口状态机
- 第一版以稳定为优先，避免做可配置化和多策略扩展
- 尽量复用现有迷你播放器窗口生命周期与命令机制，避免改动主窗口链路

## 用户交互设计

### 吸附触发
- 用户拖动迷你播放器时，保持现有自由拖拽行为
- 只有当用户松手后，才评估是否需要吸附
- 如果松手位置距离某个屏幕工作区边缘小于阈值，则将窗口吸附到最近边缘
- 如果松手位置不满足阈值，则保持当前位置，不进入贴边状态

### 自动收起触发
- 窗口吸附到边缘后，不立即收起
- 启动一次延时收起计时器，延时固定为约 1 秒
- 计时器到期时，如果窗口仍处于吸附状态且没有进入展开中的交互，则执行收起

### 收起表现
- left：窗口向左侧移出屏幕，仅保留右侧一小条可见区域
- right：窗口向右侧移出屏幕，仅保留左侧一小条可见区域
- top：窗口向上移出屏幕，仅保留底部一小条可见区域
- bottom：窗口向下移出屏幕，仅保留顶部一小条可见区域
- 第一版露出厚度固定为 10px

### 展开触发
- 当鼠标移动到可见热区，或进入收起后仍可见的窗口区域时，立即展开
- 展开时恢复到记录好的吸附后完整位置与尺寸，不重新计算新的落点
- 展开后如果鼠标继续停留在窗口上，不触发收起

### 再次收起触发
- 展开后，当鼠标离开窗口范围时，如果窗口仍然处于吸附边状态，则重新启动约 1 秒延时收起
- 如果用户在展开状态下再次拖动窗口离开边缘，则退出贴边状态，不再自动收起

## 状态模型
建议在 Rust 层为迷你播放器维护独立状态结构，例如 `MiniPlayerDockState`，字段包括：

- `docked_edge: Option<DockedEdge>`
  - 当前吸附边；未吸附时为 `None`
- `collapsed: bool`
  - 当前是否处于收起状态
- `expanded_position`
  - 展开状态下窗口左上角位置
- `expanded_size`
  - 展开状态下窗口尺寸
- `collapsed_position`
  - 收起状态下窗口左上角位置
- `collapse_deadline` 或内部计时器句柄
  - 用于管理延时收起任务
- `hover_tracking_enabled: bool`
  - 是否已进入悬停检测周期
- `last_monitor_id` 或等价监视器定位信息
  - 用于在多显示器场景下保持当前监视器上下文

边缘枚举 `DockedEdge`：
- `Left`
- `Right`
- `Top`
- `Bottom`

## 窗口几何与边缘判定

### 参考区域
- 使用当前窗口所在监视器的 `work_area` 作为主要参考边界，而不是裸屏幕尺寸
- 这样可以避开任务栏、Dock 或系统保留区，避免窗口收起后与系统 UI 冲突

### 吸附阈值
- 第一版吸附阈值固定为 24px
- 判定方式为窗口外框与工作区四条边的最短距离
- 若多个边同时满足阈值，选择距离最小的一边
- 若距离相等，优先级固定为：left > right > top > bottom，以保证结果稳定且可预测

### 吸附位置计算
吸附后，窗口完整显示时需要被调整到恰好贴边的位置：
- left：`x = work_area.left`
- right：`x = work_area.right - window.width`
- top：`y = work_area.top`
- bottom：`y = work_area.bottom - window.height`

另一轴保持用户松手时的位置，但需要夹紧在工作区范围内，避免窗口部分超出工作区：
- left/right 吸附时，对 `y` 做夹紧
- top/bottom 吸附时，对 `x` 做夹紧

## 收起与展开计算

### 收起位置
设 `peek = 10px`。

- left：`collapsed_x = work_area.left - window.width + peek`
- right：`collapsed_x = work_area.right - peek`
- top：`collapsed_y = work_area.top - window.height + peek`
- bottom：`collapsed_y = work_area.bottom - peek`

未移动的那一轴沿用吸附后的展开位置。

### 展开位置
- 展开时直接恢复到上一次记录的 `expanded_position`
- 不重新吸附，不重新做最近边计算
- 如果当前监视器工作区变化导致原位置无效，则在恢复前重新夹紧到当前工作区

## 事件与命令设计

### Rust 侧新增职责
在 [lib.rs](src-tauri/src/lib.rs) 负责：
- 初始化迷你播放器吸附状态
- 绑定迷你播放器窗口移动、悬停检查与生命周期逻辑
- 在窗口隐藏、关闭、显示时重置或同步吸附状态

在 [commands.rs](src-tauri/src/commands.rs) 负责增加必要命令，例如：
- `mini_player_check_dock_after_drag`
  - 前端在拖动结束后调用，请 Rust 判断是否应吸附并启动延时收起
- `mini_player_set_auto_hide_enabled`
  - 预留但第一版可不暴露给前端；如果实现则默认开启
- `mini_player_get_dock_state`
  - 若前端需要同步当前状态，可提供只读查询

第一版不要求完整暴露所有控制命令。最小闭环只需要一个“拖动结束后检查吸附”的入口，以及 Rust 内部自己的状态机。

### 前端触发方式
[MiniPlayerApp.vue](src/MiniPlayerApp.vue) 当前已经通过 `window_start_dragging` 发起原生拖动。第一版保持这个入口不变，但要补一个“拖动结束后的检查”机制。

推荐做法：
- 继续使用现有 `startDragging`
- 在前端监听 `mouseup` 或拖动后的稳定时机，调用新的 Tauri 命令 `mini_player_check_dock_after_drag`
- 命令进入 Rust 后，读取当前窗口真实位置并执行吸附判定

这样做的原因是：
- 前端知道用户刚完成一次鼠标拖动交互
- Rust 更适合读取窗口位置、监视器工作区与执行原生移动
- 可以避免把吸附状态机拆成前后端两半

### 前端状态同步
前端第一版不强制持有完整状态，但可以增加最小展示态：
- `data-docked-edge="left|right|top|bottom"`
- `data-collapsed="true|false"`

这些状态可以通过 Tauri 事件从 Rust 发给前端，例如新增事件：
- `mini-player:dock-state`

事件载荷建议包含：
```ts
{
  dockedEdge: 'left' | 'right' | 'top' | 'bottom' | null,
  collapsed: boolean,
}
```

前端只用于样式微调，例如：
- 收起时弱化阴影
- 不同边缘微调圆角或透明度

第一版即使不做这些样式联动，也不影响核心能力完成。

## 悬停检测策略
悬停检测是本设计里最容易不稳定的部分，必须明确边界。

### 推荐策略
- 不依赖 WebView DOM hover 判定
- 由 Rust 周期性读取全局鼠标位置，并与当前收起或展开窗口矩形做命中检测
- 仅对迷你播放器窗口启用该检测，不扩展到其他窗口

### 检测逻辑
- 当窗口处于收起状态时：
  - 如果鼠标进入当前可见热区对应矩形，则展开窗口
- 当窗口处于展开且已吸附状态时：
  - 如果鼠标离开窗口完整矩形，则启动延时收起
  - 如果鼠标重新进入窗口矩形，则取消收起计时器

### 频率
- 采用轻量轮询，例如 60ms 到 100ms 一次
- 第一版推荐 80ms，兼顾响应与资源消耗
- 轮询生命周期仅在迷你播放器显示且处于吸附/收起相关状态时启用
- 当窗口隐藏、关闭或脱离吸附状态时停止轮询

## 与现有逻辑的集成

### 迷你播放器窗口创建
[lib.rs](src-tauri/src/lib.rs) 中 `ensure_mini_player_window` 当前负责创建迷你播放器窗口。第一版保留以下现有能力不变：
- `always_on_top(true)`
- `skip_taskbar(true)`
- `transparent(true)`
- 独立窗口 label

新增的贴边自动收起逻辑只附着在该窗口上，不影响主窗口与桌面歌词窗口。

### 隐藏与关闭逻辑
当前 [MiniPlayerApp.vue](src/MiniPlayerApp.vue) 中：
- `hideWindow()` 会触发 `MINI_PLAYER_HIDE_EVENT` 并调用 `window_hide`
- `showMainWindow()` 会回发主窗口事件并隐藏迷你窗

新增规则：
- 只要迷你播放器被主动隐藏或关闭，Rust 层都要重置吸附状态与收起计时器
- 下次重新显示迷你播放器时，默认从正常展开状态开始，而不是保留上一次收起到边缘外的位置

### 原生窗口圆角/阴影
[window_chrome.rs](src-tauri/src/window_chrome.rs) 当前负责 Windows 原生圆角与阴影。

本次不修改圆角策略本身，但要接受以下视觉结果：
- 收起到边缘外时，阴影可能只在可见边处残留
- 若该效果过重，可在收起态通过前端或原生层适度减弱阴影感

这属于视觉微调，不影响第一版功能闭环。

## 错误处理与回退
- 若无法读取当前监视器或工作区，则不执行吸附，保持当前位置
- 若无法读取鼠标全局位置，则禁用自动展开/自动再收起，仅保留吸附但不收起，避免窗口卡在边缘外不可恢复
- 若收起/展开窗口位置更新失败，则立即取消当前吸附状态，恢复正常窗口模式
- 若前端未能成功发送“拖动结束后检查”命令，则迷你播放器维持现有拖动行为，不影响基础使用

这里的原则是：功能失败时退化为普通迷你窗，而不是让窗口进入不可见且不可恢复状态。

## 风险与控制

### 风险 1：拖动结束时机不稳定
如果前端 `mouseup` 与原生拖动结束的真实时机不一致，可能导致检查吸附时读到旧位置。

控制措施：
- 在前端触发检查时引入极短的稳定等待，或由 Rust 侧在接到命令后再次读取最新位置
- 不把吸附判定直接写死在 DOM 事件时刻

### 风险 2：悬停检测带来资源消耗或误判
全局鼠标轮询如果持续运行，可能造成没必要的资源消耗。

控制措施：
- 仅在窗口处于显示状态且吸附相关状态时启动轮询
- 使用固定轻量频率，不做高频刷新
- 先以矩形命中规则实现，避免过度复杂的形状判定

### 风险 3：窗口收起后难以恢复
如果收起位置计算错误，窗口可能完全离屏。

控制措施：
- 所有边缘位移都通过统一纯函数计算
- 对四边分别做单元测试与手动验证
- 失败时回退为正常展开窗口

### 风险 4：多显示器工作区切换异常
窗口横跨显示器或快速切换显示器时，边界参考可能不稳定。

控制措施：
- 第一版始终以“当前窗口所在监视器”作为唯一参考
- 不处理跨显示器半悬停等高级行为
- 如果无法可靠识别监视器，则直接不吸附

## 完成标准
满足以下条件即可认为本功能设计对应的实现完成：
- 迷你播放器支持四边吸附
- 用户松手后若足够靠边，窗口会吸附并在约 1 秒后收起
- 收起后始终保留约 10px 可见热区
- 鼠标靠近热区可展开，离开后可再次延时收起
- 主动隐藏或关闭迷你播放器时会清理吸附状态
- 前端构建与 Rust 编译检查通过
- 手动验证四边行为符合预期，没有出现窗口不可恢复的情况
