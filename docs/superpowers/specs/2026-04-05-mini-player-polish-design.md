# 迷你播放器精修设计

## 背景
当前迷你播放器已经具备基础能力：封面、标题、副标题、进度、播放控制、窗口操作、始终置顶与回到主窗口。但在实际使用中仍存在以下问题：

- 不同窗口宽度下容易显得拥挤
- 深浅色主题的层次感还不够稳定
- 按钮反馈、拖动区域与信息区之间的交互边界不够明确
- 进度条只能展示，不能直接用于定位播放进度

本次工作目标不是扩展迷你播放器功能，而是在现有结构上做一次可控的精修，让它更稳定、更清晰、更顺手。

## 目标

### 主要目标
- 保持迷你播放器轻量、紧凑、不杂乱
- 在窄宽度下仍能稳定显示，不出现重叠
- 提升深浅主题下的可读性与视觉层次
- 优化按钮、拖动热区和禁用态的交互一致性
- 支持点击进度条直接跳转到目标播放位置

### 非目标
- 不增加歌词显示
- 不增加音量控制
- 不增加多布局切换
- 不引入新的全局状态
- 不重构主播放器播放链路

## 结构设计
保留当前主结构，不重新发明布局：

- 左侧：封面区域
- 右侧顶部：标题 / 副标题 / 窗口按钮
- 右侧中部：进度条 / 当前时间 / 总时长
- 右侧底部：上一首 / 播放暂停 / 下一首

本次重点是让现有结构在不同宽度下更稳：
- 文本区始终可压缩并保持单行省略
- 按钮区固定尺寸，不参与文本挤压
- 进度区和控制区保持独立，不与标题区抢占空间

## 视觉设计

### 卡片层级
- 卡片背景继续采用半透明玻璃感，但减少厚重感
- 深色主题中降低背景浑浊度，提升前景内容清晰度
- 浅色主题中加强边框和阴影，避免控件“漂白”

### 信息层级
- 标题保持最高优先级，字号与字重略高于副标题
- 副标题保留弱化处理，但避免过低对比度
- 时间文字使用更稳定的小字号层级，保证存在感但不喧宾夺主

### 控件层级
- 主播放按钮保持最强视觉强调
- 上一首、下一首和窗口控制按钮统一为次一级
- 关闭按钮单独保留风险色提示，但不过度突出

## 交互设计

### 拖动区域
- 只保留顶部信息区作为主要拖动热区
- 按钮区域、进度区域、播放控制区域都明确标记为非拖动区
- 避免点击按钮或进度条时误触窗口拖动

### 按钮反馈
- 封面、主控按钮、窗口控制按钮都补全 hover / active / disabled 视觉反馈
- 禁用态降低透明度并保持不可交互指示
- 深浅主题下反馈风格保持一致

### 进度条交互
- 增加点击进度条跳转能力
- 点击后按点击位置换算目标时间，并同步调用播放器定位接口
- 首期不做拖拽式 seek，只做点击定位

## 自适应策略
不增加“多布局模式”，而是在单布局下加入窄宽度精修：

- 缩小水平间距与按钮间距
- 在较窄宽度下适度缩小封面和按钮尺寸
- 保持标题、副标题、时间都不换行
- 在压缩状态下优先保护标题可读性和主播放按钮尺寸

## 实现方案

### 前端
主要改动文件：
- `src/MiniPlayerApp.vue`

计划内容：
- 微调模板结构，保证拖动热区与非拖动热区分离更清晰
- 增加进度条点击定位逻辑
- 重写样式层级和自适应规则
- 优化深浅主题下的颜色、边框、阴影、hover 与 disabled 状态

### Rust
可选改动文件：
- `src-tauri/src/lib.rs`

只有在前端精修后仍然需要更合理的窗口边界时，才微调：
- 默认窗口尺寸
- 最小宽高限制
- 最大宽高限制

不改托盘、不改主窗口关闭逻辑。

## 风险与控制

### 风险
- 迷你播放器窗口较小，样式改动容易引起细节挤压
- 点击进度条定位可能与拖动窗口区域冲突
- 浅色主题可能因为透明背景出现对比度不足

### 控制措施
- 严格区分拖动区和交互区
- 所有文本继续使用单行省略策略
- 在样式上优先做减法，不增加多余控件
- 如需调整窗口最小宽度，只做小幅修改

## 验证方案

### 自动验证
- `npm run build`
- 现有前端相关测试
- `cargo check --manifest-path "E:/Polaris/music/src-tauri/Cargo.toml"`

### 手动验证重点
- 长标题歌曲是否稳定省略且不挤压按钮
- 较窄宽度下是否无重叠、无错位
- 进度条点击是否能准确跳转
- 深色与浅色主题下是否都清晰
- 拖动窗口时是否不会误触按钮与进度条

## 完成标准
满足以下条件即可认为本次精修完成：

- 迷你播放器在常见宽度范围内不出现重叠与杂乱
- 视觉层级更清晰，深浅主题都可读
- 按钮反馈与禁用态统一
- 进度条点击跳转正常工作
- 构建、测试、Rust 编译检查全部通过
