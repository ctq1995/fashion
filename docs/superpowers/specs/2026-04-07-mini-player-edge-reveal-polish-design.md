# Mini Player Edge Reveal Polish Design

## 背景

当前迷你播放器已经具备稳定的三边吸附与收起/展开能力：

- 仅支持左、右、上三边吸附隐藏
- 吸附后延迟收起，仅保留可见边
- 鼠标靠近对应边缘时重新展开
- 用户手动将展开态窗口拖离吸附范围后，不会再被强行吸回

现阶段的主要问题不再是功能缺失，而是体验偏“硬”：

- 收起后靠近边缘时，展开触发过于直接
- 鼠标快速掠过边缘时容易误展开
- 展开时视觉反馈较弱，方向感不足

本次设计目标是在不破坏当前稳定性的前提下，优化左、右、上三边的唤出节奏与视觉动画，让体验更柔和。

## 目标

本次设计要达成以下结果：

- 左、右、上三边在收起态下唤出时增加短缓冲
- 鼠标快速掠过边缘时，尽量不触发误展开
- 鼠标稳定靠近边缘时，窗口以柔和渐显方式展开
- 展开动画具备边缘方向感
- 不改变当前收起延迟、拖拽机制和底边禁用规则

## 非目标

本次不包含以下内容：

- 不恢复底边吸附与隐藏
- 不新增半展开态或预热态 UI
- 不让前端参与展开命中判定
- 不重做 mini-player 布局结构
- 不引入额外窗口或系统级原生动画机制

## 方案选型

本次采用“方案 A：短缓冲 + 定向柔和渐显动画”。

### 采用原因

- 在现有 Rust 统一控制展开逻辑的前提下，改动最小
- 可以明显减少边缘掠过时的误触
- 不需要新增中间状态或复杂事件语义
- 能在保留现有稳定性的基础上提升交互手感

### 不采用其他方案的原因

- 不采用“预热态 + 完整展开”，因为需要引入新的中间状态，回归面更大
- 不采用“只改前端动画”，因为无法解决触发节奏过硬和误触展开问题

## 总体设计

### 1. Rust 侧：收起态唤出增加 reveal buffer

当前收起态下，hover tracking 一旦判定鼠标进入边缘热区，就会立即执行展开。

本次改为：

1. 鼠标进入左/右/上对应的边缘热区
2. 不立刻展开，而是启动一次短 reveal buffer 调度
3. reveal buffer 到期后重新检查当前条件
4. 只有在条件仍成立时才真正展开窗口

这样可以把“靠近即展开”调整为“靠近并短暂停留后柔和展开”。

### 2. Reveal buffer 的重新校验条件

reveal buffer 到期后，必须重新检查以下条件：

- 当前 mini-player dock state 仍存在
- 当前 reveal session 仍为最新
- 当前仍处于 `collapsed = true`
- `docked_edge` 仍是 `left | right | top`
- 鼠标仍位于该边缘的触发热区内

只有全部满足时，才允许：

- 应用 `expanded_rect`
- 将 `collapsed` 改为 `false`
- 失效旧的 leave/collapse 调度
- 发出最新 dock-state 事件

如果任一条件不成立，则直接退出本次 reveal，不做补偿性展开。

### 3. Reveal session 设计

为避免旧 reveal 延迟任务覆盖新状态，需要在现有 dock state 中增加一类独立 session：

- `reveal_session`

它的职责与现有 `hover_session`、`collapse_session` 一致：

- 每次准备安排新的 reveal buffer 时递增
- reveal 任务运行前校验 session 是否仍匹配
- reset 时统一失效
- 展开成功或状态切换时刷新，防止旧 reveal 任务补执行

这样可以避免以下问题：

- 鼠标刚进热区就离开，旧 reveal 任务仍然展开
- 多次快速进出热区时，早期 reveal 任务覆盖晚期状态
- 用户拖动或取消 dock 后，旧 reveal 任务仍然触发展开

### 4. Reveal 调度边界

Rust 侧 reveal buffer 仅在以下条件下触发：

- 当前处于收起态
- 当前存在合法的 `docked_edge`
- 鼠标进入当前边缘对应的收起热区
- 当前还没有等价的 reveal 已完成展开

并保持以下约束：

- 不改变展开后的离开缓冲和收起延迟逻辑
- 不恢复底边热区
- 不新增第二套展开检测入口
- reveal 仍通过现有 hover tracking 入口安排

## 前端动画设计

### 1. 动画触发条件

前端动画只负责视觉表现，不负责判定是否应该展开。

在 [MiniPlayerApp.vue](src/MiniPlayerApp.vue) 中，仅在以下条件同时满足时触发 reveal 动画：

- 上一次状态是 `collapsed = true`
- 当前状态变为 `collapsed = false`
- 当前 `dockedEdge` 是 `left`、`right` 或 `top`

普通显示迷你播放器窗口、切换主题、更新播放状态等路径，不复用这套 reveal 动画。

### 2. 动画表现

按边缘方向区分轻微位移：

- 左侧展开：从左向右轻微位移进入
- 右侧展开：从右向左轻微位移进入
- 顶部展开：从上向下轻微位移进入

配合统一的透明度渐显：

- 初始轻微偏移 + 低透明度
- 结束时回到原位 + 完全可见

动画表现应保持克制，不引入明显弹跳、放大或夸张缩放。

### 3. 动画作用范围

动画仅作用于 mini-player 卡片视觉层：

- 不改变窗口最终几何尺寸
- 不改变 Rust 端真实窗口位置计算
- 不依赖额外的前端布局重排

也就是说：

- Rust 负责真正展开到 `expanded_rect`
- 前端只在视觉上补一个短暂的 reveal motion

## 参数建议

首版采用以下默认值：

- reveal buffer：`120ms`
- reveal animation duration：`180ms`
- reveal offset：`10px`
- easing：`ease-out`

参数原则：

- buffer 需要足够短，不能让用户感到迟钝
- animation 需要比 buffer 更明显，但不能拖慢操作感
- 位移幅度只提供方向感，不应造成“明显滑来滑去”的廉价感

## 状态与职责边界

### Rust 负责

- 收起态热区命中
- reveal buffer 调度与取消
- 真正窗口展开
- dock state 生命周期管理

### 前端负责

- 根据 dock-state 变化触发 reveal 动画态
- 根据当前边缘选择动画方向
- 保持其余 UI 逻辑不变

### 明确不做

- 前端不自行决定何时展开
- 前端不新增吸附热区几何判断
- Rust 不负责控制 CSS 动画细节

## 风险与规避

### 风险 1：唤出变钝

如果 reveal buffer 过长，会让用户感觉靠近边缘后响应迟缓。

规避方式：

- buffer 保持在短时范围
- 首版固定为 `120ms`
- 若手感仍偏迟，可再微调而不改架构

### 风险 2：旧 reveal 任务误展开

如果没有独立 session，快速进出热区时旧任务可能晚到并错误展开。

规避方式：

- 引入 `reveal_session`
- reveal 执行前必须校验 session + cursor + collapsed 状态

### 风险 3：前端动画影响普通显示路径

如果动画触发条件过宽，可能导致普通显示迷你窗口也带上边缘展开动画。

规避方式：

- 只在 `collapsed: true -> false` 且存在边缘信息时触发
- 普通窗口显示不走这套 reveal motion

## 测试与验证

### 自动化验证

保留现有编译与接线测试，并新增最小回归保护，确保：

- Rust 侧存在独立 reveal session / reveal buffer 调度
- 底边逻辑未被重新引入
- 前端存在基于 `dockedEdge` 的 reveal 动画分支

### 编译验证

实现后必须运行：

- `cargo check --manifest-path "src-tauri/Cargo.toml"`
- `npx vitest run "src/utils/miniPlayer.test.ts"`

### 手动验证

需要重点手动确认：

1. 左侧收起后，鼠标快速掠过边缘，不应误展开
2. 左侧收起后，鼠标稳定靠近边缘，应短缓冲后柔和展开
3. 右侧收起后，展开方向正确，应呈现从右向左的轻微位移
4. 顶部收起后，展开方向正确，应呈现从上向下的轻微位移
5. 展开后鼠标离开，仍按当前延迟规则收起
6. 用户将展开态窗口拖离吸附范围后，不应再次被自动吸回

## 实施约束

本次实现必须遵守：

- 不新增新的前端判定入口
- 不恢复底边逻辑
- 不改变当前窗口固定尺寸
- 不重写整套 mini-player 样式结构
- 不做与 reveal 无关的吸附算法重构

## 预期结果

完成后，mini-player 在左、右、上三边的收起后唤出体验应具备以下特征：

- 靠近边缘时不再生硬瞬开
- 快速掠过边缘时更少误触
- 稳定靠近时能柔和展开
- 展开动画具备明确的边缘方向感
- 不牺牲当前已修复的稳定性行为
