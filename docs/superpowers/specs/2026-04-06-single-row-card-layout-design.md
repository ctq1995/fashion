# 以“我的喜欢”为基准的共享歌曲行设计

## 背景
当前歌曲列表经历了多轮调整后，已经确认此前“强行单行塞满信息”的方向是错误的，相关改动也已回退。

用户本次重新确认的目标是：

- 以“我的喜欢”作为唯一布局基准
- 其他模块的歌曲行整体对齐“我的喜欢”
- 使用共享歌曲行组件实现统一，而不是继续在各面板各改一套
- 统一补上歌曲总长度信息
- 歌曲总长度固定放在右侧独立位，位于按钮区前面
- 本地音乐不参与这次统一

## 目标

### 主要目标
- 抽出一个共享歌曲行骨架组件，统一歌曲行的结构节奏
- 让以下模块整体对齐“我的喜欢”的歌曲行布局：
  - 搜索结果
  - 最近播放
  - 播放列表
- 在统一骨架下保留各模块必要的业务差异
- 在统一结构中稳定显示歌曲总长度
- 保持现有播放、收藏、下载、删除、加入队列等功能不变

### 非目标
- 不改本地音乐 [LocalLibraryPanel.vue](src/components/LocalLibraryPanel.vue) 的独立布局
- 不引入新的业务 store 模型
- 不重写播放、收藏、下载等业务逻辑
- 不追求一个覆盖所有场景的“万能组件”
- 不做与本次布局无关的重构

## 适用范围
本次统一覆盖以下面板：

- `src/components/FavoritesPanel.vue`
- `src/components/SearchPanel.vue`
- `src/components/HistoryPanel.vue`
- `src/components/PlaylistPanel.vue`

明确排除：

- `src/components/LocalLibraryPanel.vue`

## 设计结论
采用方案 C，但不是抽一个“全自动万能歌曲项组件”，而是抽一个“共享骨架组件”。

核心原则：
- 共享组件只统一骨架，不吞掉各模块业务逻辑
- 共享组件以“我的喜欢”当前的视觉结构作为母版
- 各模块通过 props 或 slot 注入自己的附加信息和按钮
- 总时长是统一固定信息位，不再由各模块自由摆放
- 播放列表虽然是抽屉场景，但内部歌曲项骨架仍与“我的喜欢”保持一致

## 共享组件边界

### 组件职责
共享歌曲行组件负责：
- 行容器
- 序号区
- 封面区
- 主信息区
- 播放中标签位
- 右侧总时长位
- 按钮区
- 响应式降级规则

### 不放入组件内部的内容
以下内容不在组件内部硬编码：
- 搜索模块的来源标签文案与取值逻辑
- 最近播放模块的历史附加信息文案
- 播放列表的当前项判断逻辑
- 各模块自己的按钮行为
- 各模块对副标题和附加信息的具体拼接规则

### 设计取舍
本次共享的是“结构骨架”，不是“统一业务模型”。

原因：
- 搜索、最近播放、播放列表虽然需要统一外观，但承载的信息并不相同
- 如果抽象成一个大而全的万能组件，容易再次把特殊场景强压成错误结构
- 共享骨架 + 外部注入内容，既能统一布局，又能避免再次改偏

## 统一骨架结构
共享歌曲行组件统一为以下区块：

1. 左侧基础区
   - 序号（如需要）
   - 封面

2. 中间内容区
   - 标题行：歌名 + 播放中标签
   - 副标题行：歌手 · 专辑（或模块对应基础信息）
   - 附加信息区：用于显示来源、历史信息等模块特有内容

3. 右侧固定信息区
   - 歌曲总长度

4. 最右按钮区
   - 各模块自己的操作按钮组

## 歌曲总长度规则
歌曲总长度为本次新增统一要求，规则如下：

- 所有纳入统一的模块都显示歌曲总长度
- 总长度固定放在右侧独立位
- 总长度位于按钮区前面
- 总长度不并入副标题
- 总长度不放入附加信息区
- 总长度使用统一格式化逻辑，保持视觉稳定

信息优先级：
1. 歌名
2. 总长度
3. 按钮区
4. 歌手 · 专辑
5. 播放中标签
6. 附加信息区

这意味着：
- 宽度不足时，优先压缩中间内容区
- 不允许总长度被附加信息挤掉
- 不允许按钮区被歌名和附加信息挤乱

## 各面板映射

### 我的喜欢
“我的喜欢”是本次布局母版，同时也接入共享歌曲行组件，避免出现“基准面板本身和共享组件不一致”。

映射：
- 标题行：歌名 + 播放中标签
- 副标题行：歌手 · 专辑
- 附加信息区：默认无
- 总长度：显示在右侧固定信息区
- 按钮区：播放 / 下载 / 移除收藏

### 搜索结果
映射：
- 标题行：歌名 + 播放中状态（如需要）
- 副标题行：歌手 · 专辑
- 附加信息区：来源标签
- 总长度：显示在右侧固定信息区
- 按钮区：收藏 / 加歌单 / 加队列 / 下载 / 播放

要求：
- 来源标签不再脱离整体结构单独漂移
- 搜索结果继续保留自己的操作能力
- 整行节奏与“我的喜欢”一致

### 最近播放
映射：
- 标题行：歌名 + 播放中状态（如需要）
- 副标题行：歌手 · 专辑
- 附加信息区：播放时间、播放状态等历史附加信息
- 总长度：显示在右侧固定信息区
- 按钮区：收藏 / 播放 / 下载 / 删除记录

要求：
- 历史附加信息降级为附加信息区，不再占用右侧主位
- 右侧主位统一让给总长度
- 整体骨架与“我的喜欢”一致

### 播放列表
映射：
- 标题行：歌名 + 播放中状态
- 副标题行：歌手
- 附加信息区：默认空，必要时允许极少状态信息
- 总长度：显示在右侧固定信息区
- 按钮区：播放 / 移除

要求：
- 保持播放列表抽屉场景
- 不把抽屉做成普通内容页
- 只统一内部歌曲项骨架
- 视觉节奏与“我的喜欢”一致

## 响应式策略
桌面宽度下维持统一骨架：
- 左侧基础区
- 中间内容区
- 右侧总长度位
- 最右按钮区

窄宽度下允许降级，但要求如下：
- 歌名始终优先可读
- 总长度继续保留
- 按钮仍可点击
- 附加信息可以换行、折到下方或弱化
- 不允许文本与按钮重叠
- 不允许总长度和按钮漂移成混乱布局

## 样式策略

### 结构策略
- 以“我的喜欢”的现有结构节奏为样板
- 统一行高、封面尺寸、标题层级、右侧尾部收口方式
- 共享组件输出稳定 DOM 骨架
- 各面板只提供内容与行为，不再各自维护完全不同的歌曲行结构

### 文本策略
- 歌名单行省略
- 副标题单行省略
- 附加信息区可比副标题更弱化
- 播放中标签沿用现有高亮语义，但尺寸保持紧凑

### 右侧策略
- 总长度使用稳定的独立文本位
- 按钮统一成组并固定在最右
- 不允许某个模块把时长塞进副标题，另一个模块放在按钮后面

## 实现方案
主要改动文件：
- `src/components/FavoritesPanel.vue`
- `src/components/SearchPanel.vue`
- `src/components/HistoryPanel.vue`
- `src/components/PlaylistPanel.vue`
- 新增一个共享歌曲行组件文件

计划改动：
1. 提炼“我的喜欢”的歌曲行骨架，形成共享组件
2. 让 Favorites 接入共享组件，作为基准实现
3. 让 Search 接入共享组件，并通过注入方式保留来源与操作按钮
4. 让 History 接入共享组件，并把历史附加信息降级到附加信息区
5. 让 Playlist 接入共享组件，同时保留抽屉语义和较窄宽度适配
6. 为统一组件补上总长度显示能力
7. 补充或更新测试，防止再次回到分裂结构

## 风险与控制

### 风险
- 搜索按钮较多，右侧容易拥挤
- 播放列表抽屉宽度更窄，内容压缩更明显
- 如果共享组件边界定得过大，容易再次让特殊场景失真

### 控制措施
- 共享骨架，不共享业务决策
- 先让 Favorites 接入，再逐个替换 Search / History / Playlist
- 中间内容区优先压缩，右侧总长度和按钮区保持稳定
- Playlist 只统一歌曲项，不动抽屉外层结构
- LocalLibrary 保持独立，不卷入这次改动
